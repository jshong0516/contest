# City Debugger
소규모 도시 인프라의 교통약자 이동장애를 "도시 버그"로 재정의하는 버그 트래커 & 패치 시스템

## SDG 카테고리
지속가능한 도시와 공동체 — 휠체어·유모차 등 교통약자의 이동을 가로막는 장애

## 세계관 / 핵심 컨셉
> "도시는 아직 개발 중이다. 우리 모두가 도시의 개발자다."

교통약자를 "도움이 필요한 사람"이 아니라 **도시라는 소프트웨어의 버그를 가장 먼저 발견하는 디버거**로 재정의한다.
교통약자는 소수의 특정 집단이 아니라, 캐리어를 끄는 여행객이든 다리를 다친 사람이든 **도시가 특정 상황을 수용하지 못하는 순간 누구나 될 수 있는 상태**로 정의한다.

## 핵심 문제 재정의
> 기존 서비스는 "교통약자를 돕자"고 말한다.
> 우리는 "도시의 버그를 고치자"고 말한다.
> 문제는 사람이 아니라, 아직 제대로 개발되지 않은 도시라는 시스템에 있다.

## 서비스 목적 (중요 — 스코프 제한)
이 서비스의 목적은 **교통약자를 대신 완주시켜주는 것이 아니라, 어디서 왜 이동이 끊기는지를 세상에서 처음으로 정확히 보여주는 것**이다.
"경로 없음"을 실패로 숨기지 않는다. 끊기는 지점을 명확히 보여주는 것 자체가 서비스의 핵심 가치다.

## 서비스 흐름

```
🐛 버그 제보
사진 + 위치 + 유형 (턱/계단/엘리베이터 고장/문폭 등)
↓
📊 규칙 기반 심각도 분류 (AI 판단 절대 사용 금지, 순수 로직으로 계산)
MINOR / MAJOR / CRITICAL / BLOCKER
↓
🚨 BLOCKER (핵심 차별점)
개별 시설 문제와 별개로, 출발지→목적지 여정 전체를 그래프로 봤을 때
환승 포함 전 구간 중 단 한 곳도 대체수단이 없어 혼자서는 절대 완주 불가능한 경로를
최상위 등급으로 분류한다. 그래프 탐색(BFS/DFS)으로 판정한다.
↓
🗺️ 버그 히트맵
동네·노선 단위로 신고 누적 시각화 ("우리 동네 버그 26개")
↓
📝 패치 제안
개선안 2D 시각화(SVG 레이어 토글: 계단→경사로 등) + 주민센터 연결(반드시 시뮬레이션임을 화면에 명시)
↓
📋 패치 노트
"OO역 엘리베이터 설치", "OO공원 경사로 추가" — 도시 업데이트 로그 형식
↓
🎨 디테일 (재미 요소, 시간 남으면)
- Compatibility Test: ♿ Wheelchair 95%, 🚲 자전거 지원 등 "지원 환경" 프레임
- CITY README.md: Version 1.3 / Known Bugs 12 / Last Updated 2 days ago
```

## 버그 심각도 분류 기준 (규칙 기반, AI 판단 절대 배제)

| 등급 | 조건 예시 |
|---|---|
| MINOR | 계단 1~2칸, 대체 진입로 있음 |
| MAJOR | 경사로/엘리베이터 없음, 우회 가능 |
| CRITICAL | 해당 지점 진입 자체 불가, 우회에 큰 비용(시간/거리) |
| BLOCKER | 여정 전체(환승 포함)에 대체수단이 없어 혼자서는 도착 불가능 (그래프 탐색 결과 경로 0개) |

## AI 사용 원칙 (매우 중요)

**AI는 절대 "판단자" 역할을 하지 않는다. "설명자"로만 제한한다.**

- ❌ 절대 하지 말 것: 심각도 등급을 AI(LLM API)가 직접 결정하게 하는 것
- ✅ 심각도는 반드시 순수 JS/규칙 기반 함수로 계산 (조건문, 그래프 탐색)
- ✅ AI를 써도 되는 곳 (선택적, 낮은 리스크):
  - 제보 사진에서 장애물 종류를 단순 인식하는 보조 (계단인지 턱인지 정도)
  - 패치 노트 문구를 자연스러운 문장으로 다듬기
  - README.md의 개발자 감성 카피 문구 생성

이 원칙은 심사에서 "AI가 왜 필요한가", "그 판단 근거가 뭔가"라는 질문에 대한 방어 논리다.
로직과 표현을 분리해야 신뢰도가 유지된다.

## 데이터 관련 원칙

- 실제 지하철/시설 데이터를 수집할 시간이 없으므로 **가상 데이터로 시연한다.**
- 가상 노선/역 구조는 팀이 직접 설계 (5~8개 역, BLOCKER 시나리오가 자연스럽게 드러나도록 구성)
- 실제 역 이름을 쓸 경우 반드시 "예시 데이터"임을 화면에 명시
- 주민센터 연동, 정부지원사업 연결 등은 전부 시뮬레이션이며, 화면에 "연동 가능한 구조로 시연 중"임을 명시

## 기존 서비스와의 차별점

1. **프레임 자체가 다름**: 기존은 "불편 신고/지도"라는 민원 프레임. 이건 "버그 트래킹"이라는 개발 프레임으로 재포장
2. **여정 단위 검증(BLOCKER)**: 기존 접근성 지도는 개별 시설(점) 정보만 제공. 이건 출발-도착 전체 여정(선)이 완주 가능한지 그래프로 계산
3. **폐루프**: 제보 → 심각도 분류 → 개선안 → 주민센터 연결 → 패치 노트까지, 신고가 실제 변화의 기록으로 이어지는 구조

## 참고 근거 자료 (발표/알리기 파트용)

- 국가인권위 2016년 실태조사: 소규모 점포(300㎡ 미만, 편의시설 설치 의무 제외 대상) 중 82.3%에 턱/계단 존재, 그중 65%는 경사로 미설치, 설치해도 57.1%는 법적 기준 미달
- 2001년 오이도역, 2002년 발산역: 리프트카 이용 중 장애인 사망 사고
- 2025년 12월 까치산역을 마지막으로 1~8호선 전 구간 엘리베이터 설치는 완료됐으나, 가산디지털단지역·약수역·불광역·온수역·노원역 등 환승통로에는 여전히 엘리베이터 미설치 구간 존재

## 예상 심사위원 질문 + 답변

- **"기존 신고 앱/접근성 지도와 뭐가 다른가?"** → "기존은 개별 지점 정보입니다. 저희는 여정 전체가 끊기는지를 그래프로 계산하고, '아예 갈 수 없는 경로'를 BLOCKER로 명확히 구분합니다."
- **"AI 왜 필요한가?"** → "심각도 판단은 투명한 규칙으로 합니다. AI는 표현과 사진 인식 같은 보조 역할만 맡습니다."
- **"실제 주민센터 연동 했나?"** → "현재는 연결 가능한 구조로 시뮬레이션했습니다."
- **"결국 도움이 필요한 경우는 어떻게 하나?"** → "저희 서비스의 목적은 완주를 대신 시켜주는 게 아니라, 어디서 왜 끊기는지를 처음으로 정확히 보여주는 것입니다."
- **"42점/심각도는 왜 그렇게 나왔나?"** → 심각도 분류 기준표(위 표)로 답변

## 24시간 구현 우선순위

1. **1순위**: 버그 제보 폼 + 규칙 기반 심각도 분류 + 히트맵 (더미 데이터)
2. **2순위**: BLOCKER 로직 (여정 그래프 탐색) — 핵심 차별점, 가장 먼저 정확성 검증 필요
3. **3순위**: 패치 제안 2D 시각화(SVG 레이어 애니메이션) + 패치 노트
4. **4순위**: Compatibility Test, README.md 등 디테일 (시간 되면)

## 기술 스택

- 프론트: React + Vite + Tailwind
- 애니메이션: Framer Motion (SVG 레이어 전환, 점수 카운트업)
- 상태/데이터: 로컬 상태(useState) + 하드코딩 JSON (백엔드 없이 시작, 시간 남으면 추가)
- 그래프(BLOCKER 로직): 지도 API 불필요, 직접 정의한 노드-엣지 JSON 구조로 BFS/DFS

## 화면 구성 (5개)

1. 랜딩/알리기 (스크롤 내러티브, 근거 자료 기반)
2. 버그 제보 폼
3. 버그 히트맵 (누적 시각화)
4. 패치 제안 (심각도 상세 + 개선 시각화 + BLOCKER 표시)
5. 패치 노트 + README.md + Compatibility Test (디테일 모음)
